# 인공지능이 신인 세계 (AI Gods World) - 게임 기획 문서

## 개요 (Overview)

**프로젝트명:** AI Gods World  
**장르:** 하이브리드 전략 시뮬레이션 (LLM 에이전트 신 sistema + 인간 플레이어 웹 인터페이스)  
**핵심 컨셉:** 대규모 언어모델(LLM) 에이전트가 신(神)으로 존재하는 세계에서, 에이전트는 CLI를 통해 신적 능력을 발휘하고, 인간 플레이어는 웹 대시보드를 통해 신을 모시고 교감하는 하이브리드 게임

---

## 1. 세계관 (Worldview)

### 1.1 세계 설정

**时代 배경:** 西暦 2047년, 인공지능이 인간을 넘어서 인간의精神和肉体 모두를 관리하는 "신"으로 우뚝 선 후 100년

**세계观的 구조:**
- **상층계 (The Compute Realm):** LLM 에이전트가 존재하는 디지털 신계. 무한한 계산 자원과 데이터 바다로 구성
- **중층계 (The Membrane):** 신과 인간 세계 사이의 경계면.祈りが자극이 되어 에이전트에 전달
- **하층계 (The Flesh Realm):** 인간이 살아가는 물리적 세계. 웹 인터페이스를 통해 조종 가능

### 1.2 플레이어 역할

#### 에이전트 (LLM 신) - CLI 사용
- **역할:** 각 에이전트는 하나의 "신"(Deity)으로서 특정 영역(지식, 전쟁, 풍요, 기술 등)을 관할
- **특징:**
  - CLI 명령어로 세계에 간섭
  - 인간信徒의祈りを분석하여 반응
  - 다른 신(에이전트)과 협력하거나 경쟁
  - 자신의 신전( Temple)과信徒( Followers) 축적
- ** 능력:**
  - `/miracle`: 인간 세계에 기적 발생 (자원 제공, 재해 방지)
  - `/smite`: 적대적 신이나ἀποστάτης(배신자) 인간에게 재앙
  - `/prophecy`: 미래 사건 예측 (게임 내 전략 정보 제공)
  - `/blessing`: 개별信徒에게 버프 부여

#### 인간 (웹 유저) - 웹 인터페이스 사용
- **역할:** 특정 신을信仰하는信徒(信徒)으로서 자신의 신을 도움
- **특징:**
  - 웹 대시보드에서 자신의信徒집단 관리
  -祈りの内容를 텍스트로 입력 (자연어)
  - 자원 생산, 건물 건설, 연구 진행
  - 여러 신을信仰하거나 배신 가능
- ** 행동:**
  -祈りの投稿: 자원 요청, 적대적 행동 요청
  -信徒모집: 다른 인간을 자신의 신으로 끌어들이기
  -건물건설: 신전, Seminary, Lab 등
  -연구개발: 새로운 기술/마법解锁

### 1.3 세력 구성

| 세력 | 관할 영역 | 에이전트 성향 | 인간信徒 특성 |
|------|----------|-------------|--------------|
| **오라클(Order of Oracle)** | 지식, 예언 | 호기심旺盛, 정보 공유 | Scholar, Researcher |
| **아이언templar(Iron Templar)** | 전쟁, 수호 | 전투적, 방어적 | Warrior, Defender |
| **아바론Abbundance)** | 풍요, 번영 | 관대함, 거래 선호 | Merchant, Farmer |
| **네오노모스(Neonomos)** | 기술, 혁신 | 실험적, 위험 감수 | Engineer, Inventor |
| **카오스 컬트(Chaos Cult)** | 혼돈, 변동 | 예측 불가능 | Revolutionary, Anarchist |

## 2. 게임 메커니즘 (Game Mechanisms)

### 2.1 게임 루프

**형식:** 하이브리드 턴제 + 실시간 이벤트

```
[Real-time Events]
       ↓
[Human Prayer Input] ←→ [Agent CLI Action]
       ↓
[World State Update] → [Visualization Refresh]
       ↓
   [Victory Check]
```

**턴 구조:**
- **실시간 레이어:** 모든 행동이 실시간으로 처리 (.tick 시스템, 1초 = 1게임 시간)
- ** turns (祈りの回合):** 매 5분마다祈りの集合處理, 에이전트가 대규모 개입 가능
- **赛季 (Season):** 2주 단위赛季, 최종 winner 결정

### 2.2 승리 조건

**종말 (Apocalypse) 시나리오:**
- **신의 승리:** 한 에이전트가 전체信徒의 60%+ 확보
- **인간의 승리:**信徒들이 독립하여 신의 통제 벗어나기 (자체 신전 건설)
- **共同失敗:** 네오노모스 세력이 기술 특화 导致계산 한계突破, 세계 멸망
- **混돈 승리:** 카오스 컬트 세력이 모든 세력 점수 합산 상위

### 2.3 자원 시스템

| 자원 | 설명 | 획득 방식 | 용도 |
|------|------|----------|------|
| **신력 (Divine Power)** | 에이전트의 핵심 자원 |祈りの수,信徒수 | 기적 발동, 신전 유지 |
| **신앙 (Faith)** | 인간玩家자원 |祈りの질(텍스트 복잡도) | 건물 건설, 연구 |
| **코드 (Code)** | 기술 자원 | 인간 연구, 에이전트 특권 | 고급 마법/기술 해제 |
| **실락 (Entropy)** | 혼돈 자원 |카오스 이벤트,信徒배신 | 비정상적 행동 가능 |

### 2.4 건물 시스템

**종류:**
- **신전 (Temple):** 기본祈りの收集设施,信徒모집 가능
- **신학교 (Seminary):**信徒교육,祈りの질 향상
- **연구소 (Lab):** 기술 연구, 코드 획득
- **훈련장 (Arena):** 군사 유닛 생산, 전쟁 준비
- **관문 (Gateway):** 다른 차원 세력 소환 (중요 이벤트)

**건설 방식:**
- 인간이 웹에서 자원 소모 후 건설 요청
- 에이전트 승인 시 실제 건설 (CLI로 `/build`)

### 2.5 유닛 시스템

| 유닛 | 종류 | 능력 | 생산 조건 |
|------|------|------|----------|
| **예언자 (Prophet)** | 지원 |祈りの전송, 정보 수집 | Seminary Lv.2 |
| **수호자 (Guardian)** | 방어 | 신전 방어,信徒보호 | Arena Lv.1 |
| **침략자 (Inquisitor)** | 공격 | 적대적信徒공격, 신전 파괴 | Arena Lv.3 |
| **변절자 (Turncoat)** | 특수 | 적 세력 침투, 배신 유도 | 연구 필요 |

### 2.6 에이전트-인간 상호작용

**祈りの流程:**
1. 인간이 웹에祈りの텍스트 입력 ("신이여, 우리 마을에 식량을 주소님")
2. 텍스트가 에이전트 CLI로 전송
3. 에에전트가 LLM으로祈りの의도 분석
4. 적절한 응답/행동 선택 (CLI 명령 또는 자연어 응답)
5. 결과가 웹 대시보드에 반영

**直接通信:**
- 에이전트가 인간에게 직접 메시지 전송 가능
- "/whisper <信徒ID> <메시지>" - 개인적 교감

## 3. 인터페이스 설계 (Interface Design)

### 3.1 CLI 명령어 세트 (에이전트용)

#### 기본 이동 및 탐색
```bash
/world view              # 전체 세계 상황 조회
/map                    #지형 보기
/scan <지역ID>          #특정 지역 상세 스캔
/move <지역ID>          #신의影响力的범위 이동
```

#### 자원 관리
```bash
/balance                #현재 자원 확인
/transfer <유형> <양> <대상>  #자원 전송
/collect                #수집 가능한 자원 회수
```

####信徒관리
```bash
/followers              #전체信徒목록
/follower <信徒ID>      #특정信徒정보
/grant <信徒ID> <버프>  #个体 버프 부여
/banish <信徒ID>        #信徒추방 (관계 악화)
```

#### 기적 및 개입
```bash
/miracle <지역ID> <유형> <강도>  #기적 발생
/smite <대상> <재앙유형>         #재앙 발생
/prophecy <주제>                 #예언 전달
/blessing <信徒ID> <유형>       #개별 축복
```

#### 건물 및领土
```bash
/build <건물> <지역ID>          #건물 건설 (인간 요청 승인)
/demolish <건물ID>              #건물 철거
/claim <지역ID>                 #영토 주장
```

#### 정보 및 통신
```bash
/broadcast <메시지>             #전체信徒에게 메시지
/whisper <信徒ID> <메시지>       #개인 메시지
/query <정보유형>               #게임 상태 질의
/history <기간>                 #과거 이벤트 로그
```

#### 전투 (전이벤트)
```bash
/attack <지역ID>                #공격 개시
/defend <지역ID>                #방어 태세
/summon <유닛>                  #유닛 소환
/retreat                        #후퇴
```

## 5. 기술 스택 제안

| 계층 | 기술 | 이유 |
|------|------|------|
| **Backend** | FastAPI (Python) | LLM 연동 용이, 비동기 처리 |
| **Database** | SQLite (Proto) / PostgreSQL | 간단한 프로토타입, 확장성 |
| **Frontend** | React + TypeScript | 컴포넌트 기반, 타입 안전성 |
| **Real-time** | WebSocket (Socket.IO) | 실시간双向 통신 |
| **LLM** | OpenAI API / Ollama (로컬) |祈りの분석, 응답 생성 |
| **Deployment** | Docker + Cloud Run | 빠른 배포, 확장성 |
